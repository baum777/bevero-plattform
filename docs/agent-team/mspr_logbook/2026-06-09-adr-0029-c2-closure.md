## 1. Slice summary

ADR-0029-C.2 is the first mutation-surface slice under ADR-0029-C
(Event-Economic-Rules). This slice ships a `WITH CHECK`-gated
manager-verification mutation surface for the 4 CUBE_Economic tables
(`ExclusiveRentalPolicy`, `AfterMidnightStaffRate`, `NonFoodComponent`,
`FurniturePolicy`): 2 forward-only migrations (RLS + trigger
relaxation), 3 `src/modules/cube-economic/*` file extensions, 1
route file with 4 new POST endpoints, 9 new vitest cases (19 total
in the file, up from 10), a 12-query Supabase verify-script, an
ADR section in `docs/DECISIONS.md`, and this closure MSPR. **No
existing model, migration, route, or test is modified**; the slice
is forward-only, additive, and DEMO_MODE-safe. The
Brutto/Netto-Disziplin (binding decision ADR-0029-C §5) is
preserved (the per-table whitelist excludes all `*NetCents` fields).
No LLM, no writeback, no service-role credential on user paths, no
Cockpit change.

### 1.2 Documented deviations

- **No `prisma/schema.prisma` change.** The mutation methods are
  additive to the existing service contract; the existing models
  are unchanged. The `CUBE_EconomicDatabaseClient` interface
  widens (in `cube-economic.types.ts`) to include the
  `automationDecision` / `automationRule` / `automationSuggestion`
  delegates + `$transaction` + `$executeRawUnsafe`, but this is a
  TypeScript-level change, not a Prisma schema change.

- **Per-table whitelist excludes all `*NetCents` fields.** The
  manager-verification path is constrained to
  `isActive / requiresManagerConfirmation / notes / validFrom /
  validUntil / effectiveFrom / effectiveUntil` per table. The
  Brutto/Netto-Disziplin (ADR-0029-C §5) is preserved at the DB
  layer (the CHECK constraints are unchanged). A future ADR-0029-D
  may add a separate monetary-field path with a different role
  gate (admin-only with 4-eyes principle); that is out of scope
  here.

- **Synthetic `AutomationRule` pattern.** The mutation path creates
  a per-org `AutomationRule` named
  `cube_economic_manual_verification` (`ruleType: "event"`,
  `condition: { type: "manual_detection", target: "cube_economic" }`,
  `action: { type: "manual_verification", surface: "cube_economic" }`,
  `metadata: { synthetic: true, slice: "adr-0029-c.2" }`) on demand.
  This mirrors the ADR-0029-B §Decisions §4 pattern that resolved
  the `AutomationSuggestion.ruleId` non-nullable contradiction.
  The synthetic rule is `enabled: true, version: 1`; the
  service-layer `findFirst` is non-deterministic under concurrent
  inserts (Risk ID-008), but the cost is duplicate rule rows, not
  data corruption; a future ADR-0029-D may add a `pg_advisory_xact_lock`.

- **Deterministic synthetic `AutomationSuggestion` id.** The
  service-layer uses the id
  `sug-cube-eco-<orgId-prefix>-<rowKind>-<rowId>` (sliced to 64
  chars) so subsequent verify calls on the same
  (org, rowKind, rowId) tuple reuse the same suggestion, and the
  per-event `AutomationDecision` rows are appended immutably. The
  suggestion is created with `status: "approved"` on creation
  (the manager is the approver).

## 5. Residual risks

1. **Risk ID-008 (synthetic `AutomationRule` race):** 2 concurrent
   verify calls on the same (org, rowKind, rowId) tuple both try
   to create the synthetic rule; the second create succeeds with
   a duplicate rule. Cost: duplicate rule rows, not data
   corruption. Future fix: `pg_advisory_xact_lock` (out of scope
   for this slice).

2. **Risk ID-007 (GUC leakage):** mitigated by `SET LOCAL` (not
   `SET`); the GUC is transaction-scoped and auto-reverts at
   COMMIT/ROLLBACK. A service-layer crash mid-transaction causes
   a ROLLBACK; the next transaction on the same connection starts
   with the GUC unset.

3. **Risk ID-006 (default Supabase privileges on `authenticated` /
   `anon`):** the companion REVOKE migration
   `20260609060002_revoke_cube_economic_write_grants` is already
   applied (from the ADR-0029-C promotion gate). The verify-script
   Q5 asserts the absence of write grants (PASS: 0).

4. **Risk ID-005 (verify-script textual cmd codes):** the
   verify-script uses the **textual** `pg_policies.cmd` codes
   (`'UPDATE'`, `'SELECT'`, `'INSERT'`) per the ADR-0029-B
   post-promotion correction. The single-letter codes (`'r'`,
   `'a'`, `'w'`, `'d'`) are not the form exposed by `pg_policies`
   on Supabase Postgres 15.

5. **Risk ID-004 (defense-in-depth `DO $$` block fragility):** the
   block asserts trigger **presence** (count = N), not policy
   counts. Future slices (ADR-0029-D, ADR-0029-B.2 in the parallel
   track) will widen the assertion to include their new triggers.

6. **Risk ID-002 (Brutto/Netto-Disziplin violation):** mitigated
   by the per-table whitelist (the `*NetCents` fields are NOT in
   the whitelist) + the DB CHECK constraints (unchanged). The
   service-layer rejects any non-whitelisted key with a 422 (one
   vitest case verifies this).

7. **Pre-promotion state (intentional, documented):** the
   verify-script reports 3 FAILs (Q1, Q2, Q4) because the 2
   migrations have not been applied yet. The Supabase promotion
   is owner-typed; the Implementer has not run it. **The slice is
   NOT promoted.** Cockpit must NOT consume the 4 new POST
   endpoints until the promotion gate passes (12/12 PASS).
