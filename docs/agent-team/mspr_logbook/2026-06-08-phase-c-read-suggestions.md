    - "**Role-aware filter deferred (must be flagged for owner decision).** The plan-doc §Phase C C-2 names 'role-aware: staff sees own; manager+ sees all in org.' The spec text at line 747-780 only says 'List open suggestions for the organization' with no per-role filter. The `AutomationSuggestion` table does **not** have an `assignRole` column; the `assignRole` lives on `AutomationRule.action.assignRole` (a JSON field, not queryable as a column). Implementing the plan-doc's role-aware filter would require either (a) a forward-only migration to add `assignRole` to `AutomationSuggestion` (deferred, requires its own ADR per the schema-impact rule), or (b) a per-rule JSONB filter at read time (workable but unindexed — performance risk on large suggestion sets). The slice ships the spec text (no per-role filter, staff+ can see all in the org), and the per-role filter is a follow-up ADR. The MSPR records the gap explicitly."

    - "**TypeScript intersection pattern for the fake db.** The new read methods on the `AutomationRuleDatabaseClient` are NOT on the `AutomationSuggestionTransactionClient` (which is the narrow shape the approve/reject `$transaction` callback receives). The two share the same `state` object so reads inside a transaction and reads outside see the same data. The pattern: `const db = { ...tx, automationSuggestion: { ...tx.automationSuggestion, findMany: ..., count: ... } } as unknown as AutomationRuleDatabaseClient`. The service's `this.db as unknown as Tx` cast works because the top-level db is a superset of the tx shape."

The 422 path for `expired`-filtered lists is unnecessary at this stage; expired suggestions are auto-archived by a future cron."
