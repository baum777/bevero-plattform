    - "When a query helper returns `{ error, value }`, the consumer can decide whether to render an error state, a fallback, or `null`. For a non-essential banner, `null` (no DOM) is the right graceful-degradation path: the page still works, the user just doesn't see the suggestion hint. The dashboard's `openSuggestions` field is a counter (always rendered, even if 0) which is the right pattern for a primary metric; the bar-refill banner is a hint (rendered only when meaningful) which is the right pattern for a non-essential surface."
  - gotchas:
    - "**Supabase `count` semantics in RLS-scoped queries.** `countOpenAutomationSuggestions` and `countRefillSuggestions` use `{ count: 'exact', head: true }`. The `head: true` flag tells Supabase to return only the row count, not the row data. The `exact` mode runs a real COUNT query (which can be slow on very large tables) but is the only way to get the exact count. For a bar-refill banner, the `estimate` mode would be slightly faster but may drift; the exact count is correct here because the dashboard card already uses exact."
    - "**The B-1 index `AutomationSuggestion_organizationId_status_createdAt_idx` covers the count query.** The `eq('organizationId')` + `eq('status', 'open')` + `eq('type', 'refill')` predicates all hit the index. The count is O(matches) in the index, which is fast for typical bar-refill suggestion volumes (typically 0-5 per org per day). No performance risk at the spec'd scale."
    - "**The `count: 0` graceful-degradation path matters for the empty-state.** If the rules engine is not yet running (a fresh dev project with no rule evaluations), the count is 0, and the banner correctly does not render. The bar-refill page still works as before. The banner is an enhancement, not a hard requirement."
