## 1. Slice summary

ADR-0029-B.2 is the first mutation-surface slice under ADR-0029-B
(CUBE Source-Conflict-Validator). This slice ships a
`WITH CHECK`-gated manager-resolve + manager-entry mutation surface
for the 3 CUBE tables (`CUBE_Source`, `CUBE_SourceField`,
`CUBE_Conflict`): 2 forward-only migrations (RLS + trigger
relaxation), 3 `src/modules/cube-source-conflict/*` file extensions,
1 route file with 3 new POST endpoints, 5 new vitest cases (15
total in the file, up from 10), a 12-query Supabase verify-script,
an ADR section in `docs/DECISIONS.md`, and this closure MSPR. **No
existing model, migration, route, or test is modified**; the slice
is forward-only, additive, and DEMO_MODE-safe. The
`@@unique([sourceId, fieldKey])` on `CUBE_SourceField` (binding
decision ADR-0029-B §1) is unchanged. No LLM, no writeback, no
service-role credential on user paths, no Cockpit change.

### 1.2 Documented deviations

- **No `prisma/schema.prisma` change.** The mutation methods are
  additive to the existing service contract. The
  `CUBE_SourceConflictDatabaseClient` interface widens (in
  `cube-source-conflict.types.ts`) to include the
  `automationDecision` / `automationRule` / `automationSuggestion`
  / `workflowTask` delegates + `$transaction` + `$executeRawUnsafe`,
  but this is a TypeScript-level change, not a Prisma schema
  change.

- **Synthetic `AutomationRule` pattern (3 rules, 1 per
  purpose).** The mutation paths create 3 per-org synthetic
  `AutomationRule`s on demand:
  - `cube_source_manual_resolution` (manager-resolve path)
  - `cube_source_manual_rejection` (manager-reject path)
  - `cube_source_manual_entry` (manager-entry path)

  Each has `ruleType: "event"`, `condition: { type:
  "manual_detection", target: "cube_source_conflict", purpose }`,
  `action: { type: "manual_resolution", surface:
  "cube_source_conflict", purpose }`, `metadata: { synthetic: true,
  slice: "adr-0029-b.2", purpose }`. The 3 distinct rule names
  enable future Cockpit-Display filtering (e.g. "show me all
  resolve-side mutations in the last 30 days" → `WHERE ruleId IN
  (rule-resolution)`).

- **Deterministic synthetic `AutomationSuggestion` id.** The
  service-layer uses the id
  `sug-cube-src-<orgId-prefix>-<rowKind>-<rowId>` (sliced to 64
  chars) so subsequent mutation calls on the same
  (org, rowKind, rowId) tuple reuse the same suggestion. The
  suggestion is created with `status: "open"` on creation (the
  `AutomationDecision` then closes it as approved or rejected).

- **Reject path does NOT mutate the conflict.** The
  `rejectConflict` method creates the `AutomationDecision` only;
  it does not call `cUBE_Conflict.update`. Rationale: a rejection
  is an audit-trail record (the manager reviewed the conflict and
  decided not to resolve it), not a state change. The
  `CUBE_Conflict` row's `resolvedAt` remains `null`; the conflict
  is still "open" for future re-evaluation.

- **WorkflowTask creation on resolve.** The `resolveConflict`
  method creates a `WorkflowTask` with `type: "cube.conflict_resolved"`,
  `status: "open"`, `severity: "info"`, mirroring the
  `AutomationSuggestionService.approve()` post-decision
  workflow-task creation. The Cockpit-Display layer consumes this
  in a future ADR.

## 5. Residual risks

1. **Risk ID-008 (synthetic `AutomationRule` race):** 2 concurrent
   mutation calls on the same (org, rowKind, rowId) tuple both
   try to create the synthetic rule; the second create succeeds
   with a duplicate rule. Cost: duplicate rule rows, not data
   corruption. Future fix: `pg_advisory_xact_lock` (out of scope
   for this slice).

2. **Risk ID-007 (GUC leakage):** mitigated by `SET LOCAL` (not
   `SET`); the GUC is transaction-scoped and auto-reverts at
   COMMIT/ROLLBACK.

3. **Risk ID-006 (default Supabase privileges on `authenticated` /
   `anon`):** the companion REVOKE migration
   `20260609050002_revoke_cube_write_grants` is already applied
   (from the ADR-0029-B promotion gate). The verify-script Q5
   asserts the absence of write grants (PASS: 0).

4. **Risk ID-005 (verify-script textual cmd codes):** the
   verify-script uses the **textual** `pg_policies.cmd` codes
   (`'UPDATE'`, `'INSERT'`, `'SELECT'`) per the ADR-0029-B
   post-promotion correction.

5. **Risk ID-004 (defense-in-depth `DO $$` block fragility):** the
   block asserts trigger **presence** (count = N), not policy
   counts.

6. **Risk ID-003 (PII in `winningFieldValue` or `fieldValue`):**
   `sanitizePII` regex (same as read slice) + service-layer
   500-char cap (mirrors the DB CHECK
   `CHECK (length("fieldValue") <= 500)` at
   `prisma/migrations/20260609050000_add_cube_source_conflict_tables/migration.sql`;
   binding decision ADR-0029-B §8).

7. **Risk ID-002 (already-resolved conflict overwritten):**
   mitigated by the service-layer 409 + the immutable
   `AutomationDecision` history. A re-resolve attempt returns
   409; the original resolve is preserved in the
   `AutomationDecision` audit trail.

8. **Risk ID-001 (cross-org resolve):** mitigated by the
   service-layer `findFirst({ where: { id, organizationId } })` +
   the org-scope check in the WITH CHECK policy.

9. **Risk ID-009 (stale `CUBE_Conflict.sourceIds` after a
   `CUBE_Source` hard-delete):** mitigated by the service-layer
   defensive check on read (in `listConflicts` and `getConflict`).
   The mutation path does not currently filter `sourceIds`; a
   future ADR-0029-D may add the join-table.

10. **Pre-promotion state (intentional, documented):** the
    verify-script reports 4 FAILs (Q1, Q2, Q3, Q4) because the
    2 migrations have not been applied yet. The Supabase
    promotion is owner-typed; the Implementer has not run it.
    **The slice is NOT promoted.** Cockpit must NOT consume the
    3 new POST endpoints until the promotion gate passes (12/12
    PASS).
