### 1.2 What changed in the follow-up

The initial slice commit `13dbeb8` shipped 10 file families. The Supabase promotion gate (run on 2026-06-09 against `czinchfegtglmrloxlmh`) revealed two issues that the local in-memory stub tests could not catch:

1. **Verify-script Check #5 + #6 used single-letter `pg_policies.cmd` codes** (`'r'`, `'a'`, `'w'`, `'d'`) per ADR-0029-B §3 binding. The Data-DB review's claim that `pg_policies.cmd` is `char` and stores single-letter codes was a misreading of the Postgres source. **Verified empirically against Supabase Postgres 15:** the `pg_policies` view exposes the textual form (`'SELECT'`, `'INSERT'`, `'UPDATE'`, `'DELETE'`, `'ALL'`). The 0029a verify-script's `cmd = 'SELECT'` is the **working** form, not a latent bug. The CUBE script was patched to use the textual form.

2. **Verify-script Check #8 tested for the absence of write-grants on `authenticated`.** Supabase grants the `authenticated` role default privileges (SELECT, INSERT, UPDATE, DELETE, REFERENCES, TRIGGER, TRUNCATE) on every new table in `public`. RLS denies writes by default when no policy matches, so this is **not a security risk**, but the formal privilege hygiene was loose. A follow-up migration `20260609050002_revoke_cube_write_grants` adds an explicit `REVOKE INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER ON TABLE … FROM authenticated, anon` for each of the 3 CUBE tables. Check #8 was repurposed to test the absence of `WITH CHECK` write-policies (which is the real security boundary).

### 1.3 Documented deviation

The `CUBE_SourceConflictDatabaseClient` interface uses Prisma-generated delegate names (`cUBE_Source` / `cUBE_SourceField` / `cUBE_Conflict`) instead of the brief's spelled `cubeSource` / `cubeSourceField` / `cubeConflict`. This is required for the `prisma as unknown as CUBE_SourceConflictDatabaseClient` cast at `src/app.ts` to resolve at runtime, mirroring the 0029a cast pattern at `src/modules/operational-unit/operational-unit.service.ts`. The deviation is naming-only; the behavior is identical (same `findMany` / `findFirst` methods, same argument shapes). The test stub uses the same Prisma-generated keys so the production and test code paths are byte-identical at the service boundary.

## 5. Residual risks

1. **SCHEMA-003** (partial unique on `(org, name) WHERE isActive = true`) — service-layer enforced; future ADR. The v1 unique key is the 3-column `@@unique([organizationId, name, version])`; the "only one active version per (org, name)" invariant is checked at the service layer.

2. **SCHEMA-007** (`CUBE_Conflict.sourceIds` not FK-enforced) — service-layer defensive check on read; future join-table ADR-0029-B.2. `CUBE_Conflict.sourceIds` is a `String[]` (soft reference); if a `CUBE_Source` is hard-deleted in the future, the `sourceIds` array will contain stale ids. The service layer does not currently filter these; a future ADR-0029-B.2 may add a `CUBE_ConflictSource` join table.

3. **SEED-001** (no `locationId` on `CUBE_Source` for CUBE-specificity) — v1 is org-scoped; future ADR-0029-B.2 / brand-isolation follow-up may add `locationId`.

4. **SEED-003** (ID prefix `src-cube-*` lacks org/location id) — keep `src-cube-*` for v1 (single brand); future-ADR rename path documented.

5. **VERIFY-001 (superseded by the actual finding):** the original ADR-0029-B §3 binding decision claimed `pg_policies.cmd` uses single-letter codes. **This was a misreading of the Postgres source.** The actual `pg_policies.cmd` column exposes the textual form (`'SELECT'`, `'INSERT'`, `'UPDATE'`, `'DELETE'`, `'ALL'`). The pre-existing 0029a verify-script's `cmd = 'SELECT'` is the **working** form. The CUBE verify-script was patched to match. **No 0029a follow-up is needed;** the Data-DB review's §VERIFY-001 was based on an incorrect premise. Recorded here for transparency.

6. **MSPR-DB-001 (new, identified during promotion):** the `authenticated` and `anon` Supabase roles have **default privileges** on every new table in `public` (SELECT, INSERT, UPDATE, DELETE, REFERENCES, TRIGGER, TRUNCATE). The follow-up migration `20260609050002` REVOKEs the write privileges for defense-in-depth. The 0029a RLS migration has the same issue; a follow-up migration for OperationalUnit/ServiceSlot/GroupRule is out of scope for this slice and recorded as a future task.
