## Notes on the 12-query results

- **#7 and #8 used the fallback path:** the verification script first tried to INSERT a test row, then UPDATE/DELETE it. The INSERT path was likely blocked (no INSERT policy in the live DB yet; INSERT policies are part of the ADR-0023 mutation surface). The script fell back to inspecting the trigger function body for `RAISE EXCEPTION` with `TG_OP`, which the B-2 migration's trigger function contains. This is the design contract: even if the live data state prevents a runtime test of the trigger, the database-enforced invariant is verified by the function body shape.
- **#11 caught a real risk:** the query asked `SELECT rolbypassrls FROM pg_roles WHERE rolname = 'app_runtime'`. The result was `false`, confirming the ADR-0017 `NOBYPASSRLS` stance. This is the same kind of bug I caught in ADR-0023 §RLS Plan during its review (where the text said "app_runtime bypasses RLS" but the truth is the opposite). The 12-query gate's #11 catches that class of regression.
